From `.planning/research/STACK.md` §"NEW developer-prereqs":

- Wireshark + tshark + dumpcap: `sudo dnf install wireshark wireshark-cli` (Fedora)
  / `sudo apt install wireshark tshark` (Debian/Ubuntu).
- usbmon kernel module: `sudo modprobe usbmon`; verify `/sys/kernel/debug/usb/usbmon/`
  exists.
- wireshark group: `sudo usermod -aG wireshark $USER` + logout/login for non-root capture.
- usbrply (Python, dev-time): `pipx install usbrply` OR `pip install --user usbrply`.

From `.planning/research/STACK.md` §"Test-replay infrastructure":

- Capture flow: `tshark -i usbmonN -w cap.pcapng -a duration:5` then
  `tshark -r cap.pcapng -Y 'usb.device_address == NN && usb.endpoint_address == 0x81 && usb.capdata' -T fields -e usb.capdata`.
- The pcapng files from USBPcap (Windows) and from usbmon (Linux) have different
  link-layer types (`DLT_USBPCAP` vs `DLT_USB_LINUX_MMAPPED`); usbrply handles
  both transparently.

From `.planning/research/PITFALLS.md` Pitfall 21:

- Capture USB between dongle and host (`usbmon` on the bus the dongle is plugged
  into); NEVER attempt to capture RF air.
- Wireless 2.4G dongles decrypt the link and present plaintext HID to the host —
  the USB side is exactly the layer we need.
- Filter at capture time, not at replay time: cap capture file at 50 MB; if it
  fills in \<10 seconds, the filter is wrong.

From `.planning/research/PITFALLS.md` Pitfall 17:

- Capture filter at capture time: `tcpdump -i usbmon3 -w cap.pcap 'ifindex == <devnum>'`
  or Wireshark USBPcap "Capture from this device only".
- For ak980pro / 0c45:7016: drop boot-keyboard reports
  (`usb.bInterfaceProtocol != 1`) to keep only the vendor-control interface.

From `.planning/REQUIREMENTS.md` CAPTURE-02:

- Filter shape: `usb.idVendor == 0xVVVV && usb.idProduct == 0xPPPP`.

Connected devices (from CONTEXT):

- akp03_variant_3004: 0x0300 / 0x3004
- ak980pro: 0x0c45 / 0x8009
- ajazz_24g_8k: 0x3151 / 0x5007
- microdia_dongle_7016: 0x0c45 / 0x7016

Document structure (sections, in order — minimum 120 lines total):

1. **Frontmatter + intro paragraph**

   - 1-2 paragraph framing: "This runbook documents how a developer with one
     of the 4 connected AJAZZ v1.2 devices on their bench produces sanitised
     USB-HID capture fixtures for the v1.2 milestone."
   - Cross-link to `docs/policies/capture-data-hygiene.md` (the Pitfall 17
     hygiene boundary) and `.planning/research/captures/README.md` (the
     scratch sink).
   - Bold-callout: "RAW CAPTURE FILES NEVER LEAVE YOUR LOCAL MACHINE. The
     pre-commit hook will reject them; the only thing that gets committed is
     the sanitised C++ array literal under `tests/integration/fixtures/`."

1. **Prerequisites (one-time setup)**

   - Wireshark + tshark + dumpcap install:
     - Fedora: `sudo dnf install wireshark wireshark-cli`
     - Debian / Ubuntu: `sudo apt install wireshark tshark`
   - usbmon kernel module: `sudo modprobe usbmon`; verify
     `/sys/kernel/debug/usb/usbmon/` exists (lists `0u`, `1u`, ... bus monitors).
   - wireshark group membership for non-root capture:
     `sudo usermod -aG wireshark $USER` + log out / log in for the new group
     membership to take effect. Verify with `id | grep wireshark`.
   - usbrply (Python, user-local, dev-time): `pipx install usbrply` (preferred)
     OR `pip install --user usbrply`.
   - Cross-reference STACK.md §"NEW developer-prereqs".

1. **Identify the device's USB bus + address**

   - `lsusb` first — find the (vid:pid) line for the target device.
   - `lsusb -d 0c45:8009 -v 2>/dev/null | head -20` — read `iManufacturer` /
     `iProduct` strings to confirm.
   - `lsusb -t` — read the topology tree, identify which `Bus N` the device is on.
   - The bus number maps to `usbmonN`: device on `Bus 003` → capture interface
     `usbmon3`. (Note: bus indices are 1-based; `usbmon0` is "all buses"
     concatenated, which you do NOT want — Pitfall 17 cross-cutting traffic
     leak.)

1. **Per-device capture filter table**

   - Table mapping each of the 4 connected devices to:
     - codename
     - (vid:pid)
     - Wireshark display filter: `usb.idVendor == 0x0300 && usb.idProduct == 0x3004`
     - Recommended bus (from research): use `lsusb -t` to find it; vary per machine.
     - Notes column flagging:
       - `ak980pro` (0c45:8009) + `microdia_dongle_7016` (0c45:7016) — Pitfall 17:
         filter out boot-keyboard reports with `usb.bInterfaceProtocol != 1` to
         keep only the vendor-control interface (Pitfall 20 confirms 0c45:7016
         is a SEPARATE dongle, so capture both PIDs independently).
       - `ajazz_24g_8k` (3151:5007) — Pitfall 21: cap capture file at 50 MB,
         capture only during control-channel events (DPI button press, RGB
         change), NOT during sustained movement (8 kHz polling fills buffers).
       - `akp03_variant_3004` (0300:3004) — Pitfall 22: capture one image
         upload + one brightness change + one encoder rotate; diff against
         AKP03 OSS-corpus baseline (mirajazz / opendeck-akp03 / ajazz-sdk).

   Concrete table format:

   ```
   | Codename                  | VID:PID    | Capture filter                                               | Notes                                                                              |
   | ------------------------- | ---------- | ------------------------------------------------------------ | ---------------------------------------------------------------------------------- |
   | akp03_variant_3004        | 0300:3004  | usb.idVendor == 0x0300 && usb.idProduct == 0x3004           | Pitfall 22: capture image-upload-first-chunk + brightness + encoder-rotate         |
   | ak980pro                  | 0c45:8009  | usb.idVendor == 0x0c45 && usb.idProduct == 0x8009 && usb.bInterfaceProtocol != 1 | Pitfall 17: drop boot keyboard interface. Pitfall 24: RGB transition sweep is the control-channel event window |
   | ajazz_24g_8k              | 3151:5007  | usb.idVendor == 0x3151 && usb.idProduct == 0x5007           | Pitfall 21: capture only during DPI / polling / LOD / RGB button presses; cap file at 50 MB |
   | microdia_dongle_7016      | 0c45:7016  | usb.idVendor == 0x0c45 && usb.idProduct == 0x7016 && usb.bInterfaceProtocol != 1 | Pitfall 20: confirm separate dongle by unplugging ak980pro and verifying 0c45:7016 disappearance behaviour |
   ```

1. **Capture session — step by step**

   - Step 1: identify usbmonN per §3.
   - Step 2: `tshark -i usbmon3 -w /tmp/cap.pcapng -a duration:5 -f <filter>`
     — note that the `-f` flag uses BPF capture filters; the
     `usb.idVendor == ...` filter shape is a DISPLAY filter (Wireshark dialect),
     applied after-the-fact via `-Y`. For capture-time filtering, capture broadly
     on one usbmon bus and apply the display filter when extracting bytes.
     Document this distinction explicitly — it's a common usbmon footgun.
   - Step 3: trigger the device action you want to capture (press the DPI
     button, push an image from the vendor app on a parallel boot, spin the
     encoder, etc.). Document the operator-driven event window discipline.
   - Step 4: stop the capture (the `-a duration:5` already bounds it; if you
     used `dumpcap` or interactive Wireshark, Ctrl-C).
   - Step 5: read back the per-device frames:
     ```
     tshark -r /tmp/cap.pcapng \
            -Y 'usb.idVendor == 0x0300 && usb.idProduct == 0x3004 && usb.capdata' \
            -T fields -e usb.capdata
     ```
     Verify you see the bytes you expected. If the output is empty: wrong bus,
     wrong filter, or the device action did not fire on the captured window.

1. **Sanitise + convert to hex-fixture**

   - Run usbrply:
     ```
     usbrply -j /tmp/cap.pcapng > /tmp/cap.json
     ```
     Output is JSON with one `data_out` / `data_in` entry per HID transaction.
   - Inspect `/tmp/cap.json` by eye — strip any transactions outside the
     event window you want; remove anything that isn't on the device-under-test's
     address. usbrply has already filtered to the pcapng's content, but it does
     NOT enforce VID/PID — apply the device filter at the tshark stage above.
   - Invoke the plan-09-03 converter:
     ```
     scripts/hex-to-cpparray.py /tmp/cap.json \
         --device akp03_variant_3004 \
         --capture image-upload-first-chunk \
       > tests/integration/fixtures/akp03_variant_3004/image_upload_first_chunk.h
     ```
     The `--device` value is the codename from `docs/_data/devices.yaml`; the
     `--capture` value is the human-readable label that becomes part of the
     identifier and filename. Naming convention: `image_upload_first_chunk`,
     `dpi_cycle`, `polling_rate_8000hz`, `rgb_mode_static_red`, `encoder_rotate_cw`, etc.
   - Eyeball-review the generated `.h` file. Confirm:
     - It contains ONLY the bytes you expected.
     - It contains NO keystroke-derived bytes from a boot-keyboard interface
       (catastrophic Pitfall 17 leak path).
     - The array length matches your expectation (1024 for AKP03 image chunks,
       64 for AJ-series envelope, etc.).
   - `git add tests/integration/fixtures/akp03_variant_3004/image_upload_first_chunk.h`
     — only the `.h` file. The `/tmp/cap.pcapng` + `/tmp/cap.json` stay
     ephemeral; `rm` them before walking away.

1. **What you CANNOT do (anti-features)**

   - DO NOT attempt RF-air capture with an SDR or CC2531 sniffer (Pitfall 21
     out-of-scope per CLAUDE.md "no vendor RE").
   - DO NOT run the vendor's Windows app under wine / innoextract / any Delphi
     installer extraction (CLAUDE.md hard rule: clean-room reverse engineering
     only).
   - DO NOT extract / dump / flash the dongle firmware (Pitfall 21 cross-cutting
     prohibition).
   - DO NOT add a live in-app USB sniffer UI; captures are a DEVELOPER activity,
     not a USER activity (anti-feature per STACK §"Considered-and-deferred").
   - DO NOT add telemetry uploads of captures (privacy anti-feature, same
     STACK row).
   - DO NOT commit the raw pcap (CAPTURE-01 hook blocks it; this is the
     belt-and-suspenders).

1. **Troubleshooting**

   - "Permission denied opening /dev/usbmonN" → wireshark group membership not
     active; log out/in or run `newgrp wireshark`.
   - "/sys/kernel/debug/usb/usbmon/ doesn't exist" → kernel doesn't have
     CONFIG_USB_MON enabled (rare on mainstream distros; common on minimal
     containers — capture must be done on the host).
   - "tshark -Y filter returns 0 packets" → wrong bus or device wasn't
     actively transmitting during the capture window. Re-`lsusb -t` and verify
     the bus index before re-capturing.
   - "usbrply emits empty JSON" → pcapng has zero frames matching the device
     filter; check the pcapng with Wireshark GUI to see what's actually in it.
   - "hex-to-cpparray.py errors on input shape" → the JSON has unexpected
     structure (usbrply version mismatch); check `usbrply --version` matches
     the version pinned in STACK.md (v2.1.1).

   - `docs/policies/capture-data-hygiene.md` — policy + Pitfall 17 detail.

   - `.planning/research/STACK.md` §"Test-replay infrastructure".
