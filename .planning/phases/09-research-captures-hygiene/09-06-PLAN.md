1. **Title line:** `# ARCH-05: IClockCapable::setTime wire-format outcome per device (DEFAULT VERDICT — PENDING CAPTURE CONFIRMATION)`

1. **§ Context:**
   v1.1 shipped the time-sync scaffolding (Phase 5) on the assumption that some
   AJAZZ devices expose a host-settable RTC. The v1.1 D-02 honesty contract
   guarantees that `setTime` returns `NotImplemented` and the per-row glyph
   stays at exclamation when the device cannot do it. v1.2 must answer
   definitively, per connected device, whether the RTC exists. The question
   is BINARY per device, not graduated.

   Connected devices in v1.2:

   - `akp03_variant_3004` (0x0300:0x3004) — AKP03 family Stream Dock.
   - `ak980pro` (0x0c45:0x8009) — Microdia 2.4G wireless mechanical keyboard.
   - `ajazz_24g_8k` (0x3151:0x5007) — AJ-series 8K mouse (no clock anyway).
   - `microdia_dongle_7016` (0x0c45:0x7016) — separate wireless dongle.

   The cross-research convergence on this question is unusually strong (FEATURES,
   PITFALLS, ARCHITECTURE all independently arrived at the same default verdict).

1. **§ Default Verdict:**
   **NO RTC opcode exists in any of the four reference corpora.** Specifically:

   - **mirajazz crate** (Rust binding for the AKP03 family): no setTime / setRtc
     command id; only `display`, `encoder`, `button`, `brightness`.
   - **opendeck-akp03** (Andrey V. Rust crate): same — no RTC-shaped opcode.
   - **ajazz-sdk** (vendor-published partial SDK referenced in
     `docs/protocols/streamdeck/akp03.md`): same.
   - **TaxMachine AK820 Pro corpus** (clean-room reference for ak980pro):
     full command table includes RGB modes / sleep timer / mode-list; NO
     setTime entry.

   Per-device state:

   - `akp03_variant_3004`: `hasClock=false`; `setTime` stays `NotImplemented`;
     `devices.yaml` row removes `clock` from `capabilities:` (DEVICES-05 in
     Phase 10).
   - `ak980pro`: `hasClock=false`; same demotion (DEVICES-06 in Phase 12).
   - `ajazz_24g_8k`: never advertised `clock` in v1.1; stays `false`.
   - `microdia_dongle_7016`: enters catalogue at `probed` tier with
     `capabilities: []`; clock not in scope.

   PROJECT.md "Out of Scope" row 68 ("no AJAZZ device exposes a host-settable
   RTC") is **preserved unchanged**.

1. **§ Three-witness rule (Pitfall 19) — and why it cannot be satisfied for `clock`:**
   For a capability to promote from `scaffolded` to `functional`, three witnesses
   must coexist:

   1. **Capture witness:** the vendor's native app demonstrably sends this byte
      sequence when the user clicks "Sync time" in the vendor UI. For RTC, this
      is the absence of evidence — no observed byte sequence in any captured
      session.
   1. **Round-trip witness:** the device exhibits an observable state change.
      For `clock` on AKP03: there is **no firmware-rendered LCD clock widget**
      (per `docs/protocols/streamdeck/akp03.md:113-114`). Clock-on-keyface is a
      host-rendered image upload via `display`, not firmware time. For
      `ak980pro`: the 1.14" TFT renders a clock face only when the host sends
      pixels (TaxMachine corpus confirms — no `cmd 0xXX setTime` exists).
      **Round-trip witness is structurally unavailable.**
   1. **Negative witness:** sending a deliberately wrong value produces a
      different observable state. **Cannot be tested** when round-trip witness
      is absent.

   Two of three witnesses are STRUCTURALLY unavailable. Therefore: even if
   Phase 9.x captures showed a byte sequence that LOOKED like a setTime, the
   three-witness rule would still block promotion to `functional`. The default
   verdict's strength comes from the structural impossibility, not just from
   absence in the corpora.

1. **§ Explicitly forbidden anti-feature (Pitfall 19 confirmation-bias trap):**
   **DO NOT** synthesize a fake `setSystemTimeOn` wire format from bytes that
   "look like time" in a capture (`CRT...TIM 65 e8 18 67 ...` — could be a
   little-endian Unix epoch, or could be `LIG` brightness, or `TIMer` sleep,
   or random NOP padding). Pitfall 19 documents this as the classic
   confirmation-bias trap. The Out-of-Scope table in `.planning/REQUIREMENTS.md`
   row "Synthesizing a fake setSystemTimeOn..." encodes this prohibition.

1. **§ Acceptable alternative: host-rendered TftClockWidget (v1.3+ differentiator):**
   The AK980 PRO's 1.14" TFT can display a clock face IF the host sends pixels
   through the `display` capability. This is **acceptable** as a v1.3+ feature
   tracked under DISPLAY-05 (`AK980 PRO TFT image upload via cmd 0x72`). It is
   NOT routed through `IClockCapable::setTime`; the QML / UI layer renders the
   clock face into a `QImage` and ships it as a normal image upload. The user
   sees "a clock on the screen" but the device firmware sees "an image upload."
   This honest reframing is the canonical path.

1. **§ Captures-confirmation trigger (what would flip this — vanishingly unlikely):**

   - Phase 9.x captures show, on any of the four devices, a byte sequence that:
     (a) is sent by the vendor app ONLY when the user takes a time-set action
     (capture witness positive), AND
     (b) corresponds to a verifiable device-side state change (round-trip
     witness positive — would require a firmware-rendered clock widget appearing
     for the first time), AND
     (c) responds to deliberately-wrong values with a different observable
     state (negative witness positive).
   - All three would need to hold. Per (b), this requires a clock widget to
     materialise that has never been documented in any corpus. Vanishingly
     unlikely.

   The realistic Phase 9.x outcome is the verdict stands. The finalization
   run is the explicit pro-forma confirmation that v1.2's honesty contract
   rests on captured evidence, not absence-of-evidence-as-evidence.
