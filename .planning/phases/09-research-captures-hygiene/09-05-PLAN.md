1. **Title line:** `# ARCH-04: AKP03 image-encoding pipeline location (DEFAULT VERDICT — PENDING CAPTURE CONFIRMATION)`

1. **§ Context (≥1 paragraph):**
   The AKP03 family Stream Dock backend at `src/devices/streamdeck/src/akp03.cpp`
   advertises `display` capability but the body of `setKeyImage` is currently
   `NotImplemented`. The wire format is documented in `docs/protocols/streamdeck/akp03.md`:
   60×60 JPEG `Rot0` for variant_3004 / AKP03E / AKP03; 64×64 `Rot90` for
   AKP03R rev. 2 lineage. The host needs to:

   1. Resize a `QImage` (arbitrary input) to 60×60 (or 64×64 + Rot90 for
      AKP03R r2).
   1. JPEG-encode the result at quality ~85.
   1. Chunk the encoded bytes into 1024-byte HID output reports with the
      Transfer-Done flag set on the last chunk (per Pitfall 18).
      The architectural question: WHERE does the resize + JPEG-encode + chunking
      live? Three options:

   - Option A: inside `Akp03Device::setKeyImage` directly (bloats the device class).
   - Option B: a new `ajazz_imaging` static library shared across devices.
   - Option C: `src/devices/streamdeck/src/image_pipeline.{hpp,cpp}`,
     PRIVATE-linked to `ajazz_devices_streamdeck` only.

1. **§ Default Verdict:**
   **Option C** wins for v1.2. The image pipeline lives at
   `src/devices/streamdeck/src/image_pipeline.{hpp,cpp}`:

   - PRIVATE-linked to `ajazz_devices_streamdeck` (does NOT leak to
     `ajazz_devices_keyboard` or `ajazz_devices_mouse`).
   - Reuses the existing `Qt6::Gui` dependency (`QImage` + `QImageWriter`)
     that `ajazz_devices_streamdeck` already pulls in for its v1.1 sidebar
     icon rendering.
   - Exposes a free-function-or-static-method API:
     `encodeForKey(QImage const& src, Akp03Descriptor const& desc) -> std::vector<std::uint8_t>`
     where the descriptor carries the resize target dimensions + rotation +
     JPEG quality, so the same call site handles 60×60 Rot0 and 64×64 Rot90
     by table lookup.
   - The chunking helper lives in the same file:
     `chunkForUpload(std::span<const std::uint8_t> jpeg) -> std::vector<std::span<const std::uint8_t, 1024>>`
     (or equivalent) with the `bool isLast` flag enforced at the type level
     per Pitfall 18 signature recommendation.

1. **§ Rationale:**

   - **OSS-corpus convergence (HIGH confidence):** all three independent
     reverse-engineering corpora (mirajazz crate, opendeck-akp03, ajazz-sdk
     family) encode JPEG host-side. The device firmware does NOT do
     re-encoding; it accepts raw JPEG bytes chunked at the HID layer.
   - **Smallest blast radius:** PRIVATE-link discipline keeps Qt6::Gui out
     of `ajazz_core` and the other device modules. COD-031 boundary
     preserved.
   - **No new dep:** Qt6::Gui is already on `ajazz_devices_streamdeck`'s
     PRIVATE link line for v1.1 sidebar icons.
   - **Test seam alignment:** the `image_pipeline` free functions are pure
     (no I/O); they unit-test against `MockTransport`-free Catch2 cases
     by feeding a `QImage` and asserting the byte output (matches the
     Phase 9 plan 09-04 MockTransport seam architecturally).

1. **§ Considered alternatives:**

   - **Option A (inline in `Akp03Device::setKeyImage`):** rejected. `akp03.cpp`
     is already 536 lines (per repo state 2026-05-15); inlining a multi-method
     pipeline (resize + JPEG + chunk) would push the class past the maintainable
     LOC budget and inhibit unit testing.
   - **Option B (new `ajazz_imaging` static lib):** deferred to v1.3+. The
     driving cost is the future AKP815 800×480 strip-image upload (Mirabox N3
     family), which DOES want a shared imaging surface across streamdeck +
     keyboard (the AK980 PRO TFT image upload, DISPLAY-05 — explicitly
     deferred to v1.2.x). For v1.2 with one image-pipeline consumer, Option B
     is YAGNI; promote to a separate library if/when a SECOND consumer
     materialises.

1. **§ Captures-confirmation trigger (what would flip this):**
   Phase 9.x finalization run requires a captured AKP03 variant_3004 image-upload
   sequence (one full chunked send through the BAT opcode). Three outcomes
   matter:

   - **Confirms 1024-byte chunks + 60×60 JPEG Rot0:** verdict stands as written.
   - **Reveals chunk size != 1024 or image format != 60×60 JPEG Rot0:** Option C
     still wins (pipeline location unchanged), but the descriptor parameters
     for `akp03_variant_3004` change. Update `docs/_data/devices.yaml` notes
     and `docs/protocols/streamdeck/akp03.md` accordingly; no architectural shift.
   - **Reveals firmware-side encode/decode delta requiring host-side image
     manipulation beyond Qt6's reach** (e.g. a proprietary RLE on top of
     JPEG — vanishingly low probability per OSS-corpus convergence): Option C
     no longer suffices; reconsider Option B early.

   The first two outcomes are the realistic scenarios. The third is theoretical.
