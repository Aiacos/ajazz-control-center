1. **Title line:** `# ARCH-06: Composite-HID dedup decision for DeviceRegistry (DEFAULT VERDICT — PENDING CAPTURE CONFIRMATION)`

1. **§ Context:**
   The user's machine on 2026-05-15 reports four AJAZZ devices plugged in:

> (NOTE 2026-05-20: this device — USB 0x3004 — was later firmware-confirmed to be an AKP05E, codename akp05e; see STATE.md.)

   - `0300:3004` (akp03_variant_3004)
   - `0c45:8009` (ak980pro)
   - `3151:5007` (ajazz_24g_8k)
   - `0c45:7016` (unknown — not catalogued in v1.0/v1.1)

   The unknown PID is the question: is `0c45:7016` a separate physical device
   (a SONiX/Microdia wireless dongle paired with an unidentified downstream
   input device), or is it a SECONDARY HID interface of the same physical
   AK980 PRO keyboard (which would manifest as two USB interface descriptors
   with distinct PIDs but the same physical-device parent)?

   The choice is load-bearing for `DeviceRegistry::enumerate()` (which keys on
   `(vid, pid, serial)` per v1.1 ARCH-02): if `0c45:7016` is composite-HID of
   `ak980pro`, the sidebar shows TWO rows for one physical keyboard (broken
   UX), and the v1.1 hot-plug debouncer's 300 ms trailing-edge coalescing
   inside `(vid, pid, serial)` does NOT collapse across the PID boundary.

   Pitfall 20 documents the composite-HID hypothesis and the three checks that
   would confirm it: same `/sys/bus/usb/devices/` parent path, simultaneous
   disappearance on unplug, or same hub port per `lsusb -t`.

1. **§ Default Verdict: NOT firing.**

   Composite-HID dedup is **NOT** added to `DeviceRegistry::enumerate()`.
   `0c45:7016` is treated as a separate physical device and gains a row in
   `docs/_data/devices.yaml` at `probed` tier with `capabilities: []` (Phase 13
   DEVICES-08).

1. **§ Topology evidence (live `lsusb` 2026-05-15):**
   Four independent topology facts converge on the dongle hypothesis:

   | Fact                             | `0c45:7016` (the unknown)                                           | `0c45:8009` (ak980pro 2.4G dongle)                                         |
   | -------------------------------- | ------------------------------------------------------------------- | -------------------------------------------------------------------------- |
   | USB sysfs parent                 | `usb1/1-13/1-13.1/1-13.1.2` (descendant of bus-1 port-13 hub chain) | `usb1/1-10` (bus-1 port-10, direct)                                        |
   | Link speed                       | Full-Speed 12 Mbps                                                  | (varies by port)                                                           |
   | Interface shape                  | Two boot-keyboard HID interfaces with 8-byte EPs                    | Vendor-control HID + boot-keyboard composite typical for Microdia wireless |
   | iManufacturer / iProduct strings | `iManufacturer="SONiX"` / `iProduct="USB DEVICE"`                   | distinct vendor-specific strings (`AJAZZ` / `AK980 PRO` or similar)        |

   None of Pitfall 20's three composite-HID confirmations hold:

   - Different `/sys/bus/usb/devices/` parent paths (`1-13.1.2` vs `1-10`).
   - `lsusb -t` shows them as children of DIFFERENT hub ports.
   - No simultaneous-disappearance evidence yet (unplug test deferred to
     Phase 9.x — see §finalization gate).

   This is the canonical signature of a **separate wireless 2.4GHz
   Microdia/SONiX receiver dongle**, plugged into a different port from
   ak980pro's own 2.4G dongle. The paired downstream input device (whatever
   wirelessly transmits to 0c45:7016) is unidentified — Phase 13 DEVICES-09
   stub doc documents the methodology for future SKU recognition.

1. **§ Why NOT firing (rationale):**

   - **Topology rules out the composite hypothesis** at the platform level
     (USB devicefs is the authoritative source of physical-device identity).
   - **Adding dedup infrastructure now is YAGNI:** the only consumer that
     would need it is a single physical device exposing multiple PIDs from
     the same `/sys/bus/usb/devices/` parent. No such device exists in the
     v1.2 connected set.
   - **The v1.1 ARCH-02 architecture preserved:** `DeviceRegistry::enumerate()`
     keys on `(vid, pid, serial)` unchanged. The new `0c45:7016` row coexists
     with `0c45:8009` as separate sidebar entries (correct, per topology).
   - **Pitfall 20 was a precautionary research artefact**, not a confirmed
     architectural requirement. Treating it as confirmed without topology
     evidence would have shipped complexity for no value.

1. **§ Considered alternative: Composite-HID dedup in `DeviceRegistry::enumerate`:**
   Rejected. The hypothetical implementation (Pitfall 20 §"How to avoid"):

   ```
   Group hid_device_info entries by (path-parent OR serial-if-non-empty);
   pick the interface that advertises vendor-control Usage Page (0xFF00..0xFFFF);
   surface that as the dedup'd row.
   ```

   This is correct architecture IF the topology says composite. Topology
   says it does NOT. Building it without driving evidence costs:

   - Additional 60-100 LoC in `DeviceRegistry` + 2-3 new test cases in
     `tests/integration/test_composite_hid_dedup.cpp`.
   - A new dedup edge case in the v1.1 hot-plug debouncer's coalescing logic.
   - Cross-platform parity work (Windows hidapi's `serial_number` field is
     empty on some Microdia wireless dongles, removing the dedup key).
   - Maintenance burden for a behaviour that should never fire.

   If the Phase 9.x finalization unplug test contradicts the dongle hypothesis,
   the alternative is reopened — but the conditional path is documented as
   a separate Phase 12.5 (dedup infrastructure lands BEFORE Phase 12 if
   triggered). The conditional probability is LOW per topology evidence.
