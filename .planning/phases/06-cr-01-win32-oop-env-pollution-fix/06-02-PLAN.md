\<integration_test_design>

<!-- Why a real subprocess + a Python helper: the unit test proves block content; the integration test proves end-to-end that CreateProcessW with this block actually delivers the env to a child AND keeps the parent clean. Together they close CR-01. -->

The integration test does NOT exercise `OutOfProcessPluginHost` directly (that pulls in pybind11/Python interpreter linkage indirectly). Instead it exercises **the same env-block + CreateProcessW pattern the host uses** in a focused test:

1. Build a `Win32EnvBlock` with `{L"PYTHONPATH", L"C:\\test\\fixture\\sentinel"}`.
1. Find `python3.exe` on `PATH` via `_wsearchenv_s` (or fall back to a literal `python.exe` lookup).
1. Construct command line `python3.exe <AJAZZ_FIXTURES_DIR>/print_pythonpath.py`.
1. Capture parent's `_wgetenv(L"PYTHONPATH")` into `before` (may be NULL â€” handle).
1. Spawn via `CreateProcessW` with `lpEnvironment = envBlock.data()`, `creationFlags = CREATE_UNICODE_ENVIRONMENT | CREATE_NO_WINDOW`, captured stdout pipe.
1. `WaitForSingleObject` with a 30-second timeout (CI Windows runners can be slow).
1. Read child stdout, trim whitespace.
1. Capture parent's `_wgetenv(L"PYTHONPATH")` into `after`.
1. `REQUIRE(child stdout == "C:\\test\\fixture\\sentinel")` â€” proves env block delivered to child.
1. `REQUIRE(before == after)` (handle NULLâ†’NULL case) â€” proves parent env unchanged. **THIS IS THE CR-01 REGRESSION ASSERT.**

For the concurrency assert (cross-instance race the audit flagged):

- Spawn N=2 children in parallel via `std::thread` with DIFFERENT sentinel values (`"C:\\sentinel\\one"` and `"C:\\sentinel\\two"`).
- Each thread builds its own `Win32EnvBlock` with its own sentinel and waits for its own child.
- Assert thread 1's child saw `"C:\\sentinel\\one"` AND thread 2's child saw `"C:\\sentinel\\two"` â€” no cross-pollination.
- Assert parent `_wgetenv(L"PYTHONPATH")` after both joins equals the original `before` value.
  \</integration_test_design>
