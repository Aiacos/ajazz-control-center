<objective>
Complete the KEYBOARD-01 + KEYBOARD-02 capability surface for `ak980pro`. The 20-mode firmware
RGB picker (opcode 0x13, 5-packet `START 0x18 -> MODE_BEGIN 0x13 -> DATA -> SAVE 0x02 ->
FINISH 0xF0` envelope) and the brightness(0..5)/speed(0..5) plumbing ALREADY SHIPPED in Audit-3
(`IFirmwareLightingCapable`, `LightingService.setMode`, `test_ak980_firmware_lighting.cpp`). The
genuinely missing piece is **direction (0..3)**: `buildSetRgbModeData` writes direction to byte 11,
but `setFirmwareLightingMode` and `LightingService.setMode` hardcode it to 0 — the user cannot set
it. This plan plumbs `direction` through the interface, the backend, the QML facade, and the
slider UI, and confirms the 20-mode catalogue is table-driven (KEYBOARD-01) with the range derived
from the device-reported ceilings (KEYBOARD-02, Pitfall 22).

Purpose: KEYBOARD-01/02 are mostly DELIVERED; this plan closes the one honest gap (direction) and
hardens the "range derived from descriptor, never hardcoded" contract by adding a `directionMax()`
ceiling next to the existing `brightnessMax()`/`speedMax()`. The KEYBOARD-01 spec wording says
`IRgbCapable::setMode`; the codebase implemented this via `IFirmwareLightingCapable` (a dedicated
mix-in) + the QML `LightingService.setMode` invokable. That is the same user-facing capability
under a more honest interface name — we extend the shipped interface rather than renaming it
(see planning-risk note in the SUMMARY; do not regress the existing e2e test).

Output: a `direction` parameter end-to-end (interface -> backend -> service -> QML slider), a
`directionMax()` accessor, an extended e2e test, and scale-mapped sliders that never expose the raw
0..5 / 0..3 vendor scale.
