<objective>
Deliver MOUSE-02 (8 DPI stages cycled in firmware-captured order, NOT naive +1 — Pitfall 28) and
MOUSE-03 (per-stage DPI via cmd `0x21`, per-stage colour indicator, and Lift-Off-Distance via cmd
`0x23`, each settable independently) for `ajazz_24g_8k` (`3151:5007`), with byte-level MockTransport
assertions.

Purpose: This is the DPI/LOD half of the mouse capability surface. The existing `AjSeriesMouse`
backend already exposes `setDpiStage(index, DpiStage)` and a DPI-table upload, but on the AJ159
APEX opcode map (`0x54`) — which does not match the documented 8K-mouse `0x21` opcode. The stage
count is 8 (`dpiStageCount() == 8`, field-determined; the ROADMAP's earlier 6 was a stale assumption,
corrected 2026-05-20). This plan reconciles the 3151:5007 path to the documented `0x21`/`0x23`
opcodes (keeping 8 stages), routes every encode through the `kAj24g8kEnvelope`
switch-point from plan 11-01, and pins the cycle ORDER as a named capture-sourced constant so a
naive `+1` cannot creep back in.

Output: an 8-stage, capture-ordered DPI cycle + independent per-stage DPI/colour + independent LOD
on the 3151:5007 path, with MockTransport tests asserting the documented wire bytes — all flagged
capture-pending until the Phase 9.x envelope confirmation lands.
