### Vendor architectures — three different stacks

| App                                     | Stack                                                                                                       | HID transport                                                                                   | Persistence                                                                                         | Plugin system                                                                                    |
| --------------------------------------- | ----------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| **AK980 PRO** `DeviceDriver.exe`        | MFC C++ native + `mui.dll` custom UI framework                                                              | `HidD_SetFeature` (control endpoint), `LoadLibraryA("hid.dll")` dynamic-load indirection        | **File-encrypted** SQLite (`t_profile_data` + other tables)                                         | none                                                                                             |
| **AJ159 APEX** `AJAZZ Driver(R)`        | **Electron renderer + Rust gRPC sidecar** (`iot_driver.exe` listens on `127.0.0.1:3814` plaintext, NO auth) | gRPC `sendRawFeature` → Rust `hidapi` send_feature_report                                       | `sled` 0.34.7 embedded KV at `%APPDATA%\AJAZZ Driver（R）\iot_db\`                                  | none — renderer is a grpc-web client only                                                        |
| **Stream Dock** `Stream Dock AJAZZ.exe` | **Qt 5 + CEF (Chromium Embedded)** + `SDLibrary1.dll` SDK                                                   | hidapi `hid_write` (interrupt OUT) + `hid_send_feature_report` + WinUSB bulk for larger devices | Windows Registry (`HKCU\Software\HotSpot\StreamDock\<codename>\`) + SQLite + QtKeychain for secrets | **Elgato Stream Deck v6 compatible verbatim** (13 events + 13 actions + AJAZZ `setBG` extension) |

**Architectural lesson:** our in-process `ITransport` design is
*structurally safer* than AJ159's 2-process localhost-gRPC pattern (any
local process can issue HID feature reports while the vendor daemon
runs — confirmed security regression) and *more modern* than AK980's MFC
stack. Qt6/QML is the right choice — Stream Dock's pattern validates it.

### The Elgato Stream Deck v6 compat finding (STRATEGIC)

Stream Dock's `SDPluginServer` implements the Elgato Stream Deck v6 SDK
WebSocket protocol *verbatim*:

- **Handshake**: `connectElgatoStreamDeckSocket(port, uuid, registerEvent, info)` — Elgato signature
- **13 events**: `keyDown`, `keyUp`, `willAppear`, `willDisappear`, `titleParametersDidChange`, `propertyInspectorDidAppear`, `applicationDidLaunch`, `applicationDidTerminate`, `systemDidWakeUp`, `deviceDidConnect`, `deviceDidDisconnect`, `sendToPlugin`, `sendToPropertyInspector`
- **13 actions**: `setTitle`, `setImage`, `showAlert`, `showOk`, `getSettings`, `setSettings`, `getGlobalSettings`, `setGlobalSettings`, `switchToProfile`, `openUrl`, `logMessage`, `registerPlugin`, `registerPropertyInspector`
- **+ AJAZZ extension**: `setBG` for per-key background color
- Plugins spawned as `QProcess` children via bundled `node20.exe`. There is also a parallel `SDTcpServer::startListen` for dual transport.

**Implication**: implementing `SDPluginServer` (P1 §10 above) means
**existing Elgato Stream Deck plugins run natively on our app**.
Elgato has thousands of plugins in its marketplace. Largest single
ecosystem leverage point in the entire roadmap.

### Bugs in OUR code surfaced by the RE (mostly fixed this session)

#### `aj_series.cpp` mouse backend — almost everything is wrong against AJ159

| Our const                | Vendor reality                                         | Consequence                                                              |
| ------------------------ | ------------------------------------------------------ | ------------------------------------------------------------------------ |
| `kCmdCommit = 0x50`      | `FEA_CMD_MOUSE_SET_KEYMATRIX`                          | **EVERY commit() corrupted button slot 0** → fixed P0.1 (helper removed) |
| `kCmdDpi = 0x21`         | `0x54` (omnibus packet)                                | DPI silently no-op                                                       |
| `kCmdPollRate = 0x22`    | `0x04` (single byte from `_RateToNum` lookup)          | Poll rate wrong                                                          |
| `kCmdLod = 0x23`         | byte 52 of `0x53` omnibus packet                       | LOD silently no-op                                                       |
| `kCmdButton = 0x24`      | `0x50` at byte 8 (not byte 4)                          | Button bind wrong opcode + wrong payload offset                          |
| `kCmdRgb = 0x30`         | `0x07` 8-byte light packet                             | RGB silently no-op                                                       |
| `kCmdBattery = 0x40`     | nonexistent — battery is dongle-pushed via gRPC stream | NULL battery always                                                      |
| Checksum `& 0xff` (BIT8) | `& 0x7f` (BIT7 — vendor `CheckSumType.BIT7 = 0`)       | All packets rejected by firmware                                         |

Risk currently contained because AJ159 family PIDs `0x3151:0x5008/0x4026`
are NOT registered (only `0x3151:0x5007` is, likely same wire format but
post-P0.1 degrades to silent no-op instead of data corruption). Full
rewrite scheduled as P0.5 (§3 above).

#### `proprietary_keyboard.cpp` setTime had 3 distinct bugs

1. **`pkt[10] = 0x04` hard-coded** — gohv corpus hardcodes this; matches vendor only on Thursdays. Ghidra revealed vendor writes real `wDayOfWeek` (0..6). Fixed `d590392`.
1. **`m_transport->write()` (HID Output Report)** — vendor uses `HidD_SetFeature` (control endpoint SET_REPORT). Agent B disassembly exposed the import — without it, every setTime would be silent no-op. Fixed `17abda4`.
1. **3-packet envelope** — vendor sends 4 packets (START `0x18` + PREAMBLE `0x28` + DATA + SAVE `0x02` + 100 ms sleep). Missing START left state machine in wrong mode. Fixed `17abda4`.

#### Stream Dock missing `ULEND` commit sentinel

AKP-family devices need a `CRT ULEND` (5-byte command at offsets 5..9)
after every chunked image burst. We only sent `STP` flush. Per vendor RE
annotation: "may explain occasional firmware desyncs after large image
bursts". Fixed P0.3 (AKP03 + AKP05 only; AKP153 + AKP815 propagation in §4).

### Precise technical constraints uncovered

- **AK980 PRO TFT**: 240×135 RGB565, max 140-frame GIF, 256-byte GIF header
- **AKP05 / Mirabox N4 strip**: 800×480 (not 800×100 as our backend assumed), 4 zones aligned to encoders (200×480 each)
- **Sonix SN32F299** is the shared MCU across AK820 Pro / AK980 PRO / AK680 MAX / AK35i V3 MAX — wire format `0x28` clock-sync compatible across the family
- **AJ159 USB IDs**: wired `0x3151:0x5008`, wireless `0x3151:0x4026` — NOT in our registry today (the registered `0x3151:0x5007` is a different SKU but likely same chipset family)
- **AKP05 V25 sibling codenames** — vendor has ~50 SKU rebrandings for 2025 (OMNI-STREAM, MSD-ONE/TWO/PRO/NEO, GK150, M18, H1, etc.); our `register.cpp` is missing half of them
- **AK980 PRO battery vendor poll cadence**: 15 s while wireless (matches our recommended `QTimer` cadence)
- **Stream Dock dual transport**: hidapi for smaller devices (AKP03/AKP153/AKP815/AKP03-legacy), Win32 WinUSB bulk endpoints for larger displays (AKP05/Mirabox N4)
- **`libusb-1.0.dll`** is bundled with Stream Dock but **has zero call sites in `SDLibrary1.dll`** — only used by the separate `FirmwareUpgradeTool.exe`. Main app is hidapi+WinUSB; libusb appearance is a red herring from earlier inventory.

### Meta-lessons on the RE methodology

1. **Three independent sources convergence > one strong source.** ARCH-05.1 became triple-corroborated (gohv + KyleBoyer + Agent B vendor binary) — without Agent B's disassembly of `DeviceDriver.exe` exposing the `HidD_SetFeature` import, my first setTime impl would have used `write()` (Output Report) instead of `writeFeature()` and been a silent no-op against firmware.
1. **PDB debug symbols are gold.** Stream Dock Ghidra analysis succeeded in 99 s with PDB load — decompile reads like real source (`SDDevice::sendKeyImage`, `SDPluginServer::startListen`, real class hierarchies) instead of `FUN_004xxxxx`. Without PDBs the same analysis would have taken days and produced ambiguous output.
1. **Vendor binaries hide evidence in `.lan` resource files.** AK980 PRO resource ID 827 = "Time Syns" (EN typo) / "时间校正" (CN) — proves the feature *exists* before any decompile work. Always scan UTF-16 string resources first; cheap to do, often determinative.
1. **Ghidra 12 removed Jython.** All pre-existing Python scripts (including ours from earlier sessions) had to be rewritten as Java `GhidraScript` base-class implementations. Lesson: the Java scripting layer is the post-12 standard; budget rewrite time accordingly. Reusable scripts saved as `C:\Users\unilo\reverse-eng-workdir\Extract*.java` for future runs.
1. **Cross-vendor pattern analysis is subordinate to source-level evidence.** Agent C concluded "AK980 PRO probably has no firmware RTC" by analogy to SKYLOONG / AULA / RK; Agent A found explicit source-level evidence to the contrary (gohv + KyleBoyer Rust/TS code). Always prefer source > analogy when they disagree.
1. **Advanced Installer bootstrap requires `/Extract:` flag (capital E) + admin elevation** for static extraction of the inner MSI + CAB. Then `msiexec /a` produces the file tree without installing services. Reversible, no system mutation beyond the extracted folder. Useful template for future vendor RE work.
