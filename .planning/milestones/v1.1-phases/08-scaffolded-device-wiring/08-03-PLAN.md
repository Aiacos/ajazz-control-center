**D-02 from CONTEXT.md (locked):** AKP815 counts as promotion #1 (already shipped in `62da68c`). The planner's default suggestion for #2 is **Mirabox N3 (one rev)** — broadens product line beyond AJAZZ-branded SKUs; protocol artefact is a small annotated cross-reference rather than a fresh full doc.

**Selected #2 candidate (this plan, locked):** `mirabox_n3` (rev. 1), VID:PID `0x6602:0x1002`. Not the `_rev3` (`0x6603:0x1002`) or `_n3en` (`0x6603:0x1003`) siblings — those carry the same backend and can advance once additional captures confirm them.

**Why rev. 1 specifically:** The `opendeck-akp03` catalogue uses `0x6602:0x1002` as the canonical Mirabox N3 USB pair. The two newer revs share the same protocol but exist as separate rebadge SKUs; promoting the canonical one carries the strongest signal of "Mirabox N3 family supported".

**Tier rationale (`partial` not `probed`):**

- The AKP03 backend (`akp03_descriptor` factory in register.cpp lines 83–97, already wired) handles `mirabox_n3`'s enumeration + descriptor population.
- The AKP03 protocol doc (`docs/protocols/streamdeck/akp03.md`) covers the wire format (Mirabox N3 is byte-identical to AKP03 per `[opendeck-akp03]` source-of-truth).
- "Some features work end-to-end" (Tier 2 = `partial` per D-01): key reads and basic display advertised via the shared backend; encoder rotation untested for the Mirabox rebadge specifically. This justifies `partial`, not `functional`.

**Pitfall 9 (HID Report ID byte 0):** mirabox_n3.md MUST document the byte-0 convention — same as akp03 since they share the wire format. Quote/cross-reference akp03.md.

**Pitfall 10 (reinterpret_cast on packed protocol structs):** No new C++ code is expected in this plan (the backend exists). If the executor adds any C++ for this device, it MUST use byte-wise multi-byte field writes (no `reinterpret_cast<PackedStruct*>(buffer)`). The clang-tidy rule `cppcoreguidelines-pro-type-reinterpret-cast` must pass on any new code.

**Existing register.cpp coverage (verify, do NOT add):**

- `register.cpp` line 188+ ("Mirabox N3 rebrands per `[opendeck-akp03]`") should already register the `0x6602:0x1002` PID with the `akp03_descriptor` factory and codename `"mirabox_n3"`.
- If it does NOT, this plan adds that one-line entry. If it DOES, this task is a verify-and-document operation, no C++ change.

**`feature_summary` block:** Plan 04 covers `feature_summary` rollout. This plan only sets the maturity tier and writes the protocol artefact; the per-feature breakdown lands in Plan 04.
