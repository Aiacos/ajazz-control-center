**D-01 from CONTEXT.md (locked):** Rename `status` → `maturity`. Expand vocabulary:

- `scaffolded` — descriptor + factory exist, no real device-specific code path; backend may compile but does not exercise the device.
- `probed` — device enumerates correctly, capabilities respond, basic descriptor populated; no protocol writes confirmed.
- `partial` — some features work end-to-end (e.g., key reads, basic display), but advertised capability set is incomplete or untested.
- `functional` — all advertised capabilities work in practice; tested manually by maintainer or in CI.
- `verified` — `functional` + automated CI exercise on real hardware OR sustained user-confirmed reliability across multiple maintainer reports.

**1:1 value migration:** every `status: functional` → `maturity: functional`; every `status: scaffolded` → `maturity: scaffolded`. No semantic change for existing devices; just rename. The two new mid-tiers (`probed`, `partial`) are not assigned in this plan — Plan 03 promotes specific devices.

**Maturity tier color / emoji mapping (Claude's Discretion, per CONTEXT.md):**

- `scaffolded` = 🟡 (gray/amber, existing scaffolded color)
- `probed` = 🔵 (blue, "enumerated only")
- `partial` = 🟠 (amber, "some features work")
- `functional` = 🟢 (green, existing functional color)
- `verified` = ✅ (green-with-checkmark)

**AUTOGEN block convention:** README + wiki/Home + wiki/Supported-Devices have `<!-- AUTOGEN ... -->` blocks regenerated by `scripts/generate-docs.py` (entry point is `make docs` or the pre-commit hook).

**Existing devices.yaml header structure (lines 16–21):** `statuses:` block defines emoji/label/description per status code. This block becomes `maturity_tiers:` (or stays as `statuses:` rebranded — see action below).
