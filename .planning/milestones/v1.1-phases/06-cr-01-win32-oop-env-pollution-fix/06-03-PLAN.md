\<reference_urls>

- https://learn.microsoft.com/en-us/windows/win32/api/processthreadsapi/nf-processthreadsapi-createprocessw — `lpEnvironment` block layout. Implies entries are sequentially walked, suggesting first-wins by ordinary parsing semantics, but doesn't explicitly state behaviour for duplicates.
- https://nullprogram.com/blog/2023/08/13/ — Chris Wellons's analysis of Windows env block semantics; notes empirical observation that Windows uses last-wins on duplicate keys, contradicting the implicit reading of MS docs.
  \</reference_urls>

\<probe_test_design>

<!-- Read by Task 0 to know what to add to the integration test if a duplicate-key probe doesn't already exist. -->

A minimal probe spawning a Python child with a deliberately malformed env block (TWO `PYTHONPATH` entries: `L"PYTHONPATH=C:\\\\first\\0PYTHONPATH=C:\\\\second"` plus the parent env minus PYTHONPATH plus the `\\0\\0` terminator). Sort order is preserved if both entries are kept (Win32EnvBlock would normally collapse, so this probe BYPASSES Win32EnvBlock and constructs the malformed block by hand into a `std::vector<wchar_t>` — only acceptable in a test that explicitly probes Windows's behaviour).

Child stdout will be either `C:\\first` (first-wins, MS docs implicit reading) or `C:\\second` (last-wins, nullprogram observation). Test uses `WARN()` instead of `REQUIRE()` so it's informational — the test does NOT fail either way; the value is captured for documentation. The test's tag is `[duplicate-key-probe]` so it can be filtered to the CI log section that produces the resolution artefact.
\</probe_test_design>

**Status:** Resolved (CI-observed)

## Why the question existed

Microsoft's `CreateProcessW` documentation describes the `lpEnvironment` block layout but does not explicitly state behaviour for duplicate keys. Chris Wellons's analysis (nullprogram, 2023-08-13) observes that Windows actually uses last-wins, contradicting the implicit first-wins reading of MS docs.

- MS reference: https://learn.microsoft.com/en-us/windows/win32/api/processthreadsapi/nf-processthreadsapi-createprocessw
- nullprogram analysis: https://nullprogram.com/blog/2023/08/13/

## Resolution method

Plan 06-03 added a `[duplicate-key-probe]`-tagged Catch2 test that hand-constructs a malformed env block containing two `PYTHONPATH=` entries, spawns a Python child via `CreateProcessW` with `CREATE_UNICODE_ENVIRONMENT`, and captures `os.environ.get('PYTHONPATH', '')` from the child. The test uses `WARN` (not `REQUIRE`) — its purpose is observation, not assertion.

The CI run linked above shows: `{verbatim WARN output line}`.

## Why our runtime doesn't depend on the answer

`Win32EnvBlock` (Plan 06-01) merges parent env with overrides using case-insensitive key matching. The resulting block contains exactly ONE entry per key. The duplicate-key precedence question therefore has no bearing on our runtime — by construction, our spawn never produces a block with duplicates.

The answer matters for:
1. Understanding what other `CreateProcessW` callers (e.g. third-party DLLs pre-loading env, future ad-hoc helpers in this codebase) might experience.
2. Reviewing PRs that touch Win32 child-spawn paths — if a future PR builds an env block with duplicates "for safety", it should be flagged with reference to this resolution.
3. Closing WIN32-04 contractually so the requirement can be checked off.

## Future-touch notes

If a future contributor encounters this question, they should:
- Cite this artefact and the CI evidence URL.
- NOT re-run the probe unless they suspect Windows behaviour has changed (which would be a separate ADR-worthy event).
- Use the established `Win32EnvBlock` helper for any new `CreateProcessW` call site, which sidesteps the question entirely.
