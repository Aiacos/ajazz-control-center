## 1. Executive summary

- The high-level layering is sound: `core` owns HID abstractions and registry types, device-family modules implement protocols, `plugins` embeds Python behind a PIMPL, and `app` exposes Qt/QML controllers.
- The main reliability risk is exception and race handling around background infrastructure: Linux hotplug parsing can throw from the worker thread, the hotplug callback can race with the worker, and transport statistics are mutated without synchronization.
- Device protocol parsers generally use `std::span`, bounds checks, and `std::optional`, but several outbound paths accept unbounded payloads or truncate protocol length fields while still sending the full payload.
- The app layer is still scaffold-heavy: profile load/save is not implemented, `DeviceModel` lists registered devices rather than connected devices, and QML controls are mostly not wired to backend capability models.
- Plugin hosting is correctly hidden behind a PIMPL, but public methods can be called from any thread while Python calls do not explicitly acquire the GIL, and host locks are held across Python import/dispatch.
- CMake target setup has useful warnings/sanitizer modules, but CI does not run sanitizer builds and static analysis is effectively non-gating because clang-tidy configures a missing preset and then uses `|| true`.
- Unit tests cover happy-path parsers, registry, logger, and Python SDK basics, but they lack concurrency tests, fuzz/property tests, hidapi mocks, and real capture replay fixtures.
- Documentation is broad and helpful, but several architecture docs describe a future design rather than the current implementation, especially threading, EventBus internals, exception boundaries, and plugin warning behavior.
- Tooling covers formatting and Python linting well, but missing or non-enforced C++ static analysis, SBOM/license/dependency security tools, and CI action pinning are material gaps.

## 2. Architecture review

### Strengths

- `IDevice`, capability mix-ins, `ITransport`, `DeviceRegistry`, and `EventBus` create a clear core abstraction set that device backends can implement without depending on QML or application controllers (`src/core/include/ajazz/core/device.hpp:85`, `src/core/include/ajazz/core/transport.hpp:46`, `src/core/include/ajazz/core/device_registry.hpp:23`).
- Device protocol parsing is mostly stateless and testable; streamdeck parsers return `std::optional` for malformed frames instead of throwing (`src/devices/streamdeck/src/akp03.cpp:131`, `src/devices/streamdeck/src/akp05.cpp:171`, `src/devices/streamdeck/src/akp153.cpp:172`).
- The plugin host uses PIMPL to keep pybind11 out of the public header and to isolate visibility issues (`src/plugins/src/plugin_host.cpp:31`).
- Qt-affine objects are deliberately owned by the `Application` object and hotplug callbacks are posted to the GUI thread before `DeviceModel::refresh()` (`src/app/src/application.cpp:70`, `src/app/src/application.cpp:85`).
- Logging uses an atomic level filter and serializes writes to `stderr`, which is a good low-complexity thread-safe design (`src/core/src/logger.cpp:20`, `src/core/src/logger.cpp:70`).

### Weaknesses

- Current threading docs describe one reader thread per open device and a lock-free/shared-mutex event bus, while implementation uses synchronous `poll()` methods and a `std::mutex`/`unordered_map` snapshot EventBus (`docs/architecture/THREADING.md:3`, `docs/architecture/THREADING.md:29`, `docs/architecture/THREADING.md:90`, `src/core/src/event_bus.cpp:28`).
- `Application` has hotplug marshalling, but there is no app-level code that opens devices or schedules `IDevice::poll()`; device events are therefore not connected into an end-to-end event loop (`src/app/src/application.cpp:47`, `src/core/include/ajazz/core/device.hpp:145`).
- Ownership is mostly RAII, but QObject ownership is mixed with `std::unique_ptr` parented children, which is valid only if destruction order remains as currently declared and should be documented as an intentional pattern (`src/app/src/application.cpp:27`).
- `DeviceRegistry::open()` holds the registry mutex while invoking backend factories, which unnecessarily couples registry locking to hidapi object creation and can deadlock if a factory ever calls back into registry code (`src/core/src/device_registry.cpp:55`, `src/core/src/device_registry.cpp:64`).
- Error boundaries are inconsistent: transport failures throw, parsers return optional, UI docs promise controller-level conversion to user notifications, but current controllers are stubs and most device calls can propagate exceptions (`docs/architecture/ARCHITECTURE.md:94`, `docs/architecture/ARCHITECTURE.md:96`, `src/app/src/profile_controller.cpp:17`).

## 3. C++ quality

- RAII is used well for `HidTransport` handles (`src/core/src/hid_transport.cpp:47`, `src/core/src/hid_transport.cpp:63`) and for `HotplugMonitor` teardown (`src/core/src/hotplug_monitor.cpp:310`).
- Public interfaces use `std::span` for buffers and avoid raw pointer parameters in most protocol-capability paths (`src/core/include/ajazz/core/transport.hpp:82`, `src/core/include/ajazz/core/capabilities.hpp:139`).
- `std::optional` is used appropriately for parser failures and nullable battery values, but `std::expected` is absent; many APIs throw `std::runtime_error` without structured error context (`src/core/include/ajazz/core/transport.hpp:80`, `src/devices/mouse/src/aj_series.cpp:170`).
- Header hygiene is generally good; pybind11 is hidden in `.cpp`, but `ajazz_core` publicly links `Qt6::Core`, causing core consumers to inherit Qt even though most public core headers are standard-library-only (`src/core/CMakeLists.txt:12`).
- Format-string safety is a strength: `logf()` takes `std::format_string<Args...>` and therefore rejects mismatched format strings at compile time (`src/core/include/ajazz/core/logger.hpp:65`).
- Exception safety has critical gaps around C APIs: `std::stoul` in the udev loop can throw before `udev_device_unref(dev)` and unwind out of a worker thread (`src/core/src/hotplug_monitor.cpp:104`, `src/core/src/hotplug_monitor.cpp:115`, `src/core/src/hotplug_monitor.cpp:125`).
- Some UB or narrowing risks remain: unchecked CoreFoundation type casts, unchecked `wcstoul` truncation, length-byte truncation in mouse envelopes, and 16-bit macro/image length truncation (`src/core/src/hotplug_monitor.cpp:232`, `src/core/src/hotplug_monitor.cpp:148`, `src/devices/mouse/src/aj_series.cpp:67`, `src/devices/keyboard/src/proprietary_keyboard.cpp:308`).

## 4. Concurrency

- `EventBus` is thread-safe for subscribe/unsubscribe/publish in the narrow sense: it takes a mutex-protected snapshot then invokes handlers after releasing the lock (`src/core/src/event_bus.cpp:15`, `src/core/src/event_bus.cpp:30`, `src/core/src/event_bus.cpp:33`).
- `EventBus` handlers run synchronously on the publisher thread, so any plugin/UI/blocking handler can add latency or backpressure to device I/O (`src/core/include/ajazz/core/event_bus.hpp:33`).
- `DeviceRegistry` serializes registry operations, but invoking factories under the lock increases deadlock and latency risk (`src/core/src/device_registry.cpp:55`, `src/core/src/device_registry.cpp:64`).
- `HotplugMonitor::setCallback()` writes `p_->cb` without synchronization while worker threads read the same callback reference, which is a real data race if the callback is changed after start (`src/core/src/hotplug_monitor.cpp:314`, `src/core/src/hotplug_monitor.cpp:336`).
- Linux hotplug uses a blocking `poll()` plus `eventfd` wake path, but early worker returns leave `running` true until `stop()` is called; this makes `isRunning()` ambiguous after backend initialization failure (`src/core/src/hotplug_monitor.cpp:70`, `src/core/src/hotplug_monitor.cpp:323`).
- `HidTransport` stats are plain fields updated by read/write paths and returned without synchronization, conflicting with the documented read/write split across threads (`src/core/src/hid_transport.cpp:80`, `src/core/src/hid_transport.cpp:92`, `src/core/src/hid_transport.cpp:116`, `docs/architecture/THREADING.md:44`).
- Python plugin methods claim thread-callability, but public methods call Python without explicit `py::gil_scoped_acquire`; this is fragile once dispatch or load is called off the interpreter-owning thread (`src/plugins/include/ajazz/plugins/plugin_host.hpp:46`, `src/plugins/src/plugin_host.cpp:104`, `src/plugins/src/plugin_host.cpp:130`).

## 5. Build & CI

- CMake is modular, uses target aliases, warning/sanitizer interface libraries, and reasonable presets for OS-specific debug/release builds (`cmake/Warnings.cmake:7`, `cmake/Sanitizers.cmake:6`, `CMakePresets.json:16`).
- The documentation and lint workflow refer to a `dev` preset that does not exist; current presets are OS-specific (`docs/guides/BUILDING.md:48`, `README.md:150`, `CMakePresets.json:16`).
- CI builds only release presets across Linux/macOS/Windows; sanitizer-enabled `linux-debug` is present but not run as a CI matrix item (`.github/workflows/ci.yml:30`, `CMakePresets.json:20`).
- Coverage runs only on pushes, uploads an artifact, and has no minimum coverage gate (`.github/workflows/ci.yml:100`, `.github/workflows/ci.yml:141`).
- Third-party dependencies pulled with FetchContent and Flatpak git sources are tag-pinned, not commit/SHA-pinned (`CMakeLists.txt:50`, `CMakeLists.txt:59`, `tests/CMakeLists.txt:2`, `packaging/flatpak/io.github.Aiacos.AjazzControlCenter.yml:26`).
- Stable and nightly release workflows package artifacts but do not sign artifacts, generate provenance, generate checksums, or attach SBOMs (`.github/workflows/release.yml:137`, `.github/workflows/nightly.yml:160`).

## 6. Testing

- Parser tests cover key happy paths and some malformed/edge cases, such as short AKP frames and out-of-range AKP05 encoders (`tests/unit/test_akp05_protocol.cpp:96`).
- Capture replay is currently a fabricated 16-byte array despite comments claiming physical-device fixtures in `tests/integration/fixtures/` (`tests/integration/test_capture_replay.cpp:7`, `tests/integration/test_capture_replay.cpp:24`).
- No test exercises concurrent subscribe/unsubscribe/publish on `EventBus`; existing tests are single-threaded fan-out and unsubscribe cases (`tests/unit/test_event_bus.cpp:17`, `tests/unit/test_event_bus.cpp:35`).
- There are no hidapi mocks or fake `ITransport` injection points for most concrete devices; constructors directly call `makeHidTransport()` (`src/devices/streamdeck/src/akp03.cpp:208`, `src/devices/mouse/src/aj_series.cpp:91`).
- Python SDK tests cover invalid JSON fallback, but not non-object JSON, very large settings payloads, action-id grammar, or malicious plugin metadata (`python/ajazz_plugins/tests/test_plugin_api.py:55`, `python/ajazz_plugins/__init__.py:140`).
- Missing fuzz/property tests for HID frame parsers, profile JSON, plugin manifests, and mouse/keyboard packet builders leave the most byte-oriented code under-tested.

## 7. Documentation & DX

- The repo has strong developer-facing docs, including architecture pages, protocol notes, build guides, wiki material, Doxygen comments, and a security policy (`docs/architecture/ARCHITECTURE.md:92`, `SECURITY.md:22`).
- Build-from-source ergonomics are undermined by stale `dev`/`release`/`coverage`/`clang` preset documentation versus actual OS-specific preset names (`docs/guides/BUILDING.md:56`, `CMakePresets.json:16`).
- Threading documentation must be treated as aspirational until updated: it describes reader threads, GIL guards, background logger flushing, and EventBus internals that are not implemented (`docs/architecture/THREADING.md:13`, `docs/architecture/THREADING.md:29`, `docs/architecture/THREADING.md:75`, `docs/architecture/THREADING.md:90`).
- Plugin documentation promises a third-party warning chip that no code path appears to implement in the current app layer (`docs/architecture/PLUGIN-SYSTEM.md:104`, `src/app/qml/Main.qml:36`).
- The profile schema/persistence story is incomplete in code: core has a writer and stub reader, while the app controller's load/save methods are stubs (`src/core/src/profile.cpp:161`, `src/app/src/profile_controller.cpp:17`, `src/app/src/profile_controller.cpp:21`).

## 8. Tooling

- Pre-commit covers whitespace, YAML/JSON/TOML checks, clang-format, ruff, cmake-format/lint, yamllint, mdformat, and typos (`.pre-commit-config.yaml:33`, `.pre-commit-config.yaml:59`, `.pre-commit-config.yaml:69`).
- `.clang-tidy` is comprehensive, enabling bugprone, cert, clang-analyzer, concurrency, cppcoreguidelines, modernize, performance, portability, and readability checks (`.clang-tidy:2`).
- clang-tidy is not actually enforced in CI due to the missing preset and `|| true` in the run step (`.github/workflows/lint.yml:116`, `.github/workflows/lint.yml:125`).
- No cppcheck, IWYU, OSV/Trivy/Grype, SBOM, or license-compliance tooling is configured; repo search found no references to those tool names in workflows, pre-commit, CMake, or pyproject.
- Link checking is warning-only (`fail: false`), which is acceptable for early docs but should become gating for release branches (`.github/workflows/lint.yml:162`).

## 9. Performance

- `EventBus::publish()` copies an `unordered_map` of subscription tokens to `std::function` handlers on every event; this is simple and safe but can be expensive on high-frequency encoder/mouse paths (`src/core/src/event_bus.cpp:28`, `src/core/src/event_bus.cpp:31`).
- `DeviceModel::refresh()` resets the whole model on every hotplug event and always returns `connected=false`, causing avoidable QML invalidation and misleading UI (`src/app/src/device_model.cpp:41`, `src/app/src/device_model.cpp:58`).
- Device image paths stream arbitrary payload sizes in 512-byte chunks without rejecting oversized images, so a large profile image can produce long synchronous HID write bursts (`src/devices/streamdeck/src/akp153.cpp:360`, `src/devices/streamdeck/src/akp05.cpp:462`).
- QML delegates use hard-coded colors/sizes and fragile parent traversal; this increases binding churn and makes component reuse difficult (`src/app/qml/DeviceList.qml:41`, `src/app/qml/KeyDesigner.qml:22`, `src/app/qml/RgbPicker.qml:20`).
- Python dispatch is synchronous and host-lock protected, so a slow action blocks other plugin operations and can affect input latency (`src/plugins/src/plugin_host.cpp:115`, `src/plugins/src/plugin_host.cpp:130`).
