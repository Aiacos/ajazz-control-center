## Wire protocol

`[ajazz-sdk]` declares AKP815 as **v1 API** (`is_v1_api() == true`),
which is the same framing as AKP153 (512-byte packets, `CRT` prefix,
3-byte ASCII command word at bytes 5..7). All AKP153 opcodes apply
unchanged:

| Command                         | Bytes 5..7       | Same as                               |
| ------------------------------- | ---------------- | ------------------------------------- |
| `DIS` вЂ” display init            | `0x44 0x49 0x53` | AKP153                                |
| `LIG` вЂ” brightness              | `0x4C 0x49 0x47` | AKP153                                |
| `BAT` вЂ” key image               | `0x42 0x41 0x54` | AKP153 (but 100Г—100 instead of 85Г—85) |
| `CLE` вЂ” clear                   | `0x43 0x4C 0x45` | AKP153                                |
| `STP` вЂ” flush                   | `0x53 0x54 0x50` | AKP153                                |
| `LOG` вЂ” boot logo / strip image | `0x4C 0x4F 0x47` | AKP153 (800Г—480 instead of 854Г—480)   |
| `HAN` вЂ” sleep                   | `0x48 0x41 0x4E` | AKP153                                |

### Input reports

Same shape as AKP153 (single transition frame per press, byte 9 = key
index 1..15). If the firmware emits release events independently, the
implementation will need to recognise an extra byte 10 polarity вЂ” to be
confirmed with a capture.

## Catalog mapping (consumer-side)

The upstream Mirabox "Space" plugin catalog uses **its own short device
codes** that the fetcher in `src/app/src/streamdock_catalog_fetcher.cpp`
translates. The current code maps `"N4" в†’ "akp815"`, which is **wrong**:

- `N4` = 10-key Plus-class device в†’ maps to **`akp05`** (or a new
  `mirabox_n4` codename once we settle naming).
- `AKP815` = 15-key v1-API device в†’ its upstream code is unknown today.
  When we obtain a real catalog row tagged as AKP815, that mapping table
  must be amended.
