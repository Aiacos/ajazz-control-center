> Reference implementation. AKP153E (China) and AKP153R (regional rev) share

## Wire protocol

Captured against the vendor "StreamDock" utility on Windows 11 and
re-implemented in `src/devices/streamdeck/src/akp153.cpp` from the byte
descriptions below. Cross-checked against `[ajazz-sdk]` and `[zcube]`.

### Framing

| Field                     | Bytes                   |
| ------------------------- | ----------------------- |
| Header prefix `CRT`       | 0..2 (`0x43 0x52 0x54`) |
| Padding                   | 3..4 (`0x00 0x00`)      |
| 3-byte ASCII command word | 5..7                    |
| Padding                   | 8..9 (`0x00 0x00`)      |
| Payload                   | 10..511 (zero-padded)   |

All output and input reports are exactly **512 bytes** (`[ajazz-sdk]` calls
this the "v1 API"; the AKP03 family uses 1024-byte packets — see the
sister doc).

### Output reports (host → device)

Bytes 5..7 carry the command. The full opcode table cross-references
`[ajazz-sdk]/protocol/codes.rs`:

| Command                        | Bytes 5..7       | Payload encoding                                                  |
| ------------------------------ | ---------------- | ----------------------------------------------------------------- |
| `DIS` — display init           | `0x44 0x49 0x53` | —                                                                 |
| `LIG` — set brightness         | `0x4C 0x49 0x47` | byte 10 = percent (0..100)                                        |
| `BAT` — image chunk announce   | `0x42 0x41 0x54` | bytes 10..11 = big-endian JPEG size; byte 12 = 1-based key index  |
| `STP` — flush / stop           | `0x53 0x54 0x50` | —                                                                 |
| `CLE` — clear keys / show logo | `0x43 0x4C 0x45` | bytes 10..11 = tag pair (table below)                             |
| `LOG` — set boot logo (v1)     | `0x4C 0x4F 0x47` | bytes 8..15 = `0x00 0x12 0xC3 0xC0 0x01 0x..` magic + size header |
| `HAN` — sleep                  | `0x48 0x41 0x4E` | —                                                                 |
| `CONNECT` — keep-alive         | ASCII `CONNECT`  | —                                                                 |

### `CLE` tag pairs

| `byte 10` | `byte 11` | Effect                           |
| --------- | --------- | -------------------------------- |
| `0x00`    | 1..15     | Clear a single key               |
| `0x00`    | `0xFF`    | Clear all keys                   |
| `0x44`    | `0x43`    | Restore vendor logo (ASCII "DC") |

### Image upload

1. Host writes the `BAT` header packet carrying the **big-endian** JPEG
   length and the 1-based key index.
1. Host streams the JPEG payload in 512-byte chunks, last chunk
   zero-padded.
1. Device acknowledges with an input report whose first 7 bytes are
   `41 43 4B 00 00 4F 4B` ("ACK\\0\\0OK") — see `RESPONSE_ACK_OK` in
   `[ajazz-sdk]`. The backend serialises uploads per key; sending a new
   image before the ACK lands corrupts the target.

The JPEG must be 85×85, encoded at quality ≈ 90, with **90° rotation and
horizontal+vertical mirroring** baked in — `[ajazz-sdk]` describes this as
`ImageRotation::Rot90 + ImageMirroring::Both`. The current image-pipeline
TODO ("phase 2") must apply that transform before the `BAT` chain.

### Boot logo

The full 854×480 image is uploaded after a `LOG` header. Payload is
chunked just like a key image, but the magic word `0x12 0xC3 0xC0 0x01` at
offsets 8..11 of the header is **firmware-version dependent**: AKP153R
uses the shorter `LOG 0x00 0x00` header (`REQUEST_CMD_LOGO_IMAGE_V2`).
Older firmware versions on the AKP153 may reject the v2 header and need
the v1 form — our code must probe and fall back.

### Input reports (device → host)

```
byte 0..8 : 0x00
byte 9    : key index (1..15), or 0x00 for "no event"
byte 10+  : 0x00 (key-state implied by transition)
```

The firmware emits exactly **one** frame per transition with no
press/release discriminator at byte 10. This is the same hardware
limitation `[companion]` documents for "some Stream Dock models". Our
backend should diff successive frames to expose synthetic press + release
edges — today it always reports `pressed = true`, which is the open bug
already in §1 of the in-tree audit (see `gsd-debug` if revived).

### ACK

```
'A' 'C' 'K' 0x00 0x00 'O' 'K' 0x00 ...
```

Returned after each image upload completes (per-key, per-logo, per-text).

## Edge cases and quirks

- The device **silently drops** frames whose first three bytes are not
  `CRT`. Some reverse-engineering tools use this to probe firmware version
  by counting how many corrupt frames yield ACKs.
- Sending a new image to a key before its previous upload was ACK'd
  corrupts that key until the next `CLE`/reupload sequence.
- Brightness values > 100 are clamped by firmware. Our backend clamps in
  software too so tests stay deterministic.
- Frames shorter than 16 bytes are not legal HID reports for this device
  and must be discarded silently by the parser (`SEC-008`).
- The 854×480 LCD strip can be used as a **wallpaper / branding panel**;
  it is not directly addressable for live data unless the host re-uploads
  the boot logo. Treat it as static for now.

## Time sync

**Status:** scaffolded — not yet implemented.

The AKP153 firmware does not currently expose a host-settable RTC over HID
(vendor recon `d5616ef` confirmed the Stream Dock SDK has no time events).

The application's `TimeSyncService::setSystemTimeOn` calls
`IClockCapable::setTime()` on `Akp153Device`, which returns
`TimeSyncResult::NotImplemented` and emits a `WARN` log once per process
lifetime via `std::call_once(s_warned_akp153, ...)` (Pitfall 14 — no log spam).

The day a wire format is captured (vendor-app reverse-engineering or
firmware static analysis), only this backend's `setTime()` body changes —
the UI / service / capability advertisement / per-row glyph / Settings toggle
all stay. See
[`docs/superpowers/specs/2026-05-13-time-sync-design.md`](../../superpowers/specs/2026-05-13-time-sync-design.md)
for the design contract and
[`docs/superpowers/plans/2026-05-13-time-sync.md`](../../superpowers/plans/2026-05-13-time-sync.md)
for the implementation plan.
