# AJAZZ AKP153 / Mirabox HSV293S — Wire Protocol

This document captures the AKP153 HID protocol as consumed by our backend in `src/devices/streamdeck/src/akp153.cpp`. The AKP153E (China variant) shares the same wire protocol behind a different USB PID.

> The reference captures were gathered with Wireshark + USBPcap on a Windows 11 VM running the vendor's "StreamDock" utility. Every command below is cross-verified against [ZCube's published notes](https://gist.github.com/ZCube/430fab6039899eaa0e18367f60d36b3c) and re-implemented in-house.

## Identification

| Property    | Value                                 |
| ----------- | ------------------------------------- |
| Vendor ID   | `0x0300`                              |
| Product ID  | `0x1001` (AKP153), `0x1002` (AKP153E) |
| HID usage   | Vendor-defined, interface #0          |
| Packet size | 512 bytes (zero-padded)               |

## Key layout

```
 -----------------------------
| 0d | 0a | 07 | 04 | 01 | 10 |
|----|----|----|----|----|----|
| 0e | 0b | 08 | 05 | 02 | 11 |
|----|----|----|----|----|----|
| 0f | 0c | 09 | 06 | 03 | 12 |
 -----------------------------
```

Values `0x10`–`0x12` are non-interactive icons. Our backend exposes keys 1–15 to the UI and skips 16–18 until the use case is clarified.

## Output reports (host → device)

All output reports share a common 10-byte header:

```
byte 0 : 'C' (0x43)
byte 1 : 'R' (0x52)
byte 2 : 'T' (0x54)
byte 3 : 0x00
byte 4 : 0x00
byte 5..7 : 3-letter command code
byte 8..9 : 0x00 0x00
```

| Command        | Bytes 5..7 | Encoding of the rest                                             |
| -------------- | ---------- | ---------------------------------------------------------------- |
| Set brightness | `L I G`    | byte 10 = percentage (0..100)                                    |
| Set image      | `B A T`    | byte 10..11 = big-endian JPEG size · byte 12 = 1-based key index |
| Stop / flush   | `S T P`    | (no payload)                                                     |
| Clear          | `C L E`    | byte 10..11 = tag pair (see below)                               |

### Clear tag pairs

| `byte10` | `byte11` | Effect                     |
| -------- | -------- | -------------------------- |
| `0x00`   | 1..15    | Clear single key           |
| `0x00`   | `0xff`   | Clear all keys             |
| `0x44`   | `0x43`   | Clear and show vendor logo |

### Image upload

The first 512-byte frame is the `B A T` header above. The JPEG payload (image encoded at 85×85 pixels, quality ≈ 90) is then chunked into 512-byte frames padded with zeros. The device ACKs the final frame with an `A C K O K …` input report.

## Input reports (device → host)

Every input report is 512 bytes, usually mostly zeroes.

### Key event

```
byte 0..8 : 0x00
byte 9    : key index (1..15)
byte 10+  : 0x00
```

The AKP153 emits a single frame per press-and-release transition. Our backend treats each frame as a `KeyPressed` event and expects the profile engine to debounce long-holds separately. Plain "released" state is inferred from a second identical frame; per-key state is kept inside the backend.

### ACK

```
'A' 'C' 'K' 0x00 0x00 'O' 'K' 0x00 ...
```

Returned after an image upload completes.

## Edge cases and quirks

- The device silently ignores frames whose first three bytes are not `CRT`; this is used by some reference clients to probe firmware version indirectly.
- Sending a new image before receiving the ACK for the previous upload corrupts the target key. The backend serializes image uploads per key.
- Brightness values > 100 are clamped by the firmware, but our backend clamps them explicitly for deterministic tests.
