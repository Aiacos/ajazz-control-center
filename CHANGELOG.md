### Changed / Hardened

- Property Inspector security hardening pass (precedes M5 plugin-host wiring): every per-plugin `QWebEngineProfile` now has a `PIUrlRequestInterceptor` installed at the same point `PropertyInspectorController::loadInspector` constructs the profile. The interceptor is scoped to the absolute Property Inspector directory (parent of `htmlAbsPath`) and gates every sub-resource fetch. Allow/deny logic lives in pure C++ at `src/app/src/pi_url_policy.{hpp,cpp}` so it is unit-testable without QtWebEngine: file:// is allowed only inside the PI dir (with `..` traversal blocked via `QDir::cleanPath`-based comparison and a trailing-slash guard against suffix-look-alike directories), https:// only to the Phase 1 CDN allowlist (`cdn.jsdelivr.net`, `unpkg.com` â€” pinned in a `static constexpr std::array<std::string_view, 2>` so `git grep kPiHttpsCdnAllowlist` shows the entire trust surface), qrc / blob / data are passed through for Qt internals (QWebChannel ships at `qrc:///qtwebchannel/qwebchannel.js`), and http:// + every other scheme is rejected as defence-in-depth even though `LocalContentCanAccessRemoteUrls` is already off. Each deny logs `AJAZZ_LOG_WARN("plugin-pi", ...)` with the plugin UUID, the request URL and a fixed reason tag (`scheme-blocked` / `http-blocked` / `file-outside-pi-dir` / `https-not-in-allowlist` / `malformed`) so a security review can grep the noise from one bad plugin out of the rest. `$SD.openUrl(url)` is now wired through `QDesktopServices::openUrl` after an https-only validation that rejects `javascript:`, `file:`, `mailto:`, `http:`; every call is logged at INFO with the plugin UUID + URL so audit logs see plugin-initiated browser opens. The per-plugin first-call confirmation prompt is the next step and is flagged with a grep-able `// TODO(pi-prompt)` in `src/app/src/pi_bridge.cpp`. A developer aid PI page at `resources/dev/pi/index.html` exercises the full `$SD` surface (setSettings / getSettings / setGlobalSettings / getGlobalSettings / sendToPlugin / setTitle / setImage / openUrl with allowed-and-blocked URLs / logMessage) plus listens for `connected` / `didReceiveSettings` / `didReceiveGlobalSettings` / `sendToPropertyInspector`, so the bridge can be smoke-tested manually without a live plugin host. 12 new unit tests under `tests/unit/test_pi_bridge.cpp` pin the policy contract: file:// inside-vs-outside the PI dir, https:// in-vs-not-in the allowlist (including suffix-attack hosts like `cdn.jsdelivr.net.attacker.example`), http:// blocked, qrc / blob / data allowed, exotic schemes rejected, malformed URLs rejected, the openUrl policy (https allow, javascript/file/mailto/http reject), and a pin-down test on `kPiHttpsCdnAllowlist` so adding a CDN entry is a visible CI change.
